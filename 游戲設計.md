# 《未然？已然。》Game Design Document v0.2
*Working Title | Single-player foundation, multiplayer TBD*

---

## 1. 核心定位

**一句話定位：** 活在《未然？已然。》的世界裡，以第六人的身份見證霆與然的故事。

**玩家身份：** 同屆新生，剛覺醒本源的新鮮人。不是主角，不是救世主。是一個有自己生命軌跡的人，剛好跟主角團在同一個時間點出現在同一個世界裡。

**核心體驗：**
- 本源系統有機生長，沒有攻略可查，沒有人告訴你你是什麼
- 然記得你，霆記得你，他們對你的印象由你的行為決定
- 主線劇情照原著走，你是見證者，但你的存在會悄悄影響他們
- 死亡有真實代價，世界不等你

**玩家與主線的關係邊界**

敘事節點固定——主線劇情的關鍵場景與結局是原著的一部分，不會因玩家行為而改變。玩家影響的是抵達這些節點的路徑與過程中的細節。

不觸碰主線的情況下，玩家就是這個世界裡的另一個人。可以完全不與主角團產生交集，可以加入不同勢力，可以走一條與主線毫無關聯的人生軌跡。然與霆存在於這個世界裡，但不是玩家必須完成的內容。

## 2. 世界觀摘要

基於小說《未然？已然。》的完整世界觀設定。

**三大勢力：**
- Yggdrasil（北歐神話體系，隱性反派，追求標準化）
- Gaia（希臘神話體系，西方議會政府）
- 盤古（中國神話體系，中央集權帝國）

勢力衝突存在於世界背景層，不直接干涉玩家日常——除非玩家主動介入。

**能力體系：**
三層架構——本源（Root Concept）→ 詮釋（Interpretation）→ 術式（Derived Skills）
詳見第5節。

## 3. 開場與Tutorial

### 3.1 開場動畫
- 麒麟補完戲份：科普本源、詮釋、術式的世界機制與偽科學邏輯
- Cut scene：入學章霆與然相遇片段（原著場景）
- 結束後進入玩家控制頁面

### 3.2 朔源儀式（本源覺醒）
- 玩家觸碰水晶，世界變成黑白
- 看到隱約的方向感/印象，不是文字，不是標籤
- 這個impression就是本源的seed，永遠不會被明確告知
- 系統記錄這個seed用於後續LLM審核的一致性檢查

### 3.3 Tutorial流程
1. 自由時間：嘗試感知本源，開發第一個詮釋方向（第一個術式）
2. 課堂段落：隨機教師科普戰鬥系統，實戰示範
3. 分組：玩家被分配到然與霆同組
4. 實戰練習：新手教程包裝成小組對練，然持續當小太陽
5. 課程結束：自由活動開放，tutorial結束

## 4. 玩家角色設計

### 4.1 身份
- 同屆新生，背景自訂（有限選項）
- 本源randomly seeded，玩家只知道一個模糊方向
- 無預設名字，無預設性格

### 4.2 成長系統
- 無等級制
- 成長 = 詮釋深化 + 術式解鎖 + 熟練度累積
- 熟練度體現為：前後搖縮短、精準度提升、消耗降低
- 死亡懲罰：熟練度歸零、裝備消失、詮釋路徑保留（本源不變，已解鎖詮釋節點保留）

### 4.3 與主角團的關係
- 由Civilizism後端驅動
- 然與霆對玩家的印象有機生長，基於實際互動記憶
- 無固定好感度數值，無UI顯示——關係深度體現在他們對你說話的方式、記得的事、危機時刻的反應

## 5. 本源能力系統

### 5.1 三層架構

| 層次 | 說明 | 性質 |
|------|------|------|
| 本源（Root Concept） | 覺醒時隨機seed，永遠隱藏 | 固定，唯一 |
| 詮釋（Interpretation） | 玩家主動提出，LLM審核 | 動態，可深化 |
| 術式（Derived Skills） | 詮釋確立後deterministic生成 | 固定schema |

### 5.2 詮釋提出流程

1. 玩家以文字輸入詮釋方向（「既然我有《引力》與《斥力》，那我要開發《賦磁》來讓非金屬也能被吸引/排斥」）
2. LLM檢查：這個詮釋是否與已確立的節點有合理的概念連接？
3. 通過理論審核後：隨機成功率決定是否真正覺醒
4. 成功：術式schema生成，存入角色資料，變為deterministic
5. 失敗：不告知原因（理論不可行 vs. 這次失敗，玩家自行判斷）

**審核範例：**
- 已有節點「引力」「斥力」，提出「賦磁」（讓非金屬獲得磁性）→ 概念連接成立，LLM通過
- 「熱能」→ 已有節點「Agitation」，無直接連接：需玩家補充中間節點
  - 玩家補充「Agitation + 分子運動 = 熱能」：LLM驗證分子運動概念連接成立，通過
- 「重力」→ 從「電磁力」直接跳過去：節點跨度過大，LLM無法建立連接，拒絕
  - 除非玩家能說明所有中間節點且LLM認可每一步

### 5.3 術式Schema系統

詮釋確立後，LLM生成對應的術式schema，之後完全deterministic，不再調用LLM。

**Schema結構：**
```json
{
  "skill": "magnetism",
  "type": "field",
  "target": "non-living",
  "params": {
    "radius": 3.0,
    "intensity": 500,
    "latency": 1.2,
    "cost": 40
  },
  "physics_effect": {
    "modify": "magnetic_force",
    "delta": "+intensity per second"
  }
}
```

**執行邏輯：**
```javascript
skill(x, y, proficiency) {
  radius = min(r, schema.radius * proficiency),
  latency = schema.latency * (1 - proficiency),
  cost = schema.cost * (1 - proficiency)
}
```

**非物理效果的處理（heat、gravity方向改變等）：**
透過修改引擎側的材質係數或物理參數實現，而非直接傷害數值：

```json
{
  "wood": {
    "mass": 200,
    "heat_resistance": 0.3,
    "burning_point": 300,
    "burning_rate": 0.8,
    "friction": 0.6
  }
}
```

術式 heat field 作用於 wood：`temperature += intensity × (1 - heat_resistance)` per second，到達 burning_point 後點燃，並以 burn_rate 持續減少 mass。

### 5.4 節點圖系統
- 已確立的詮釋 = 節點
- 新詮釋必須能說明與既有節點的概念連接
- 路徑越長（節點越多）：cost = Σ 各節點消耗，成功率 = Π 各節點成功率
- 已審核通過的節點與邊 = deterministic，不再經過LLM

### 5.5 戰鬥哲學
- 前中期：物理對抗為主，術式輔助
- 後期：術式作為佈局與心理博弈工具
- 最優打法永遠是讓環境打人，不是直接用術式打人

## 6. NPC系統（Civilizism後端）

### 6.1 架構
- 所有重要NPC由Civilizism驅動
- 記憶系統：基於實際互動生成記憶
- 人格：會因互動而緩慢漂移

### 6.2 主角團特殊處理
- 然與霆使用完整Civilizism loop
- 對玩家的印象基於真實互動記憶，無腳本好感度
- 主線劇情節點到達時，他們的反應會受到與玩家關係深度影響
- API等級：使用高品質LLM（主角團）vs 輕量模型（路人NPC）

### 6.3 視野外NPC
- 玩家視野外：簡化loop，不跑完整perception-behavior cycle
- 進入視野：activate完整loop
- Zone-based loading，對應物理引擎的active zone

## 7. 物理引擎系統

### 7.1 設計哲學
- 術式作為初始條件輸入，物理引擎負責所有後續結果
- 無傷害數值，傷害 = 物理動量/結構破壞的自然結果

### 7.2 材質系統
- 每種材質有完整物理係數：mass、heat_resistance、burning_point、burning_rate、friction 等
- 資料來源：Wikipedia已知材質特性，一次性建表

### 7.3 活體與非活體的邊界
- **非活體**：術式完整生效，字面物理效果
- **活體**：術式直接作用時效果極度削弱（能量場機制）
- **邊界判定**：術式的直接作用對象是否包含活體
  - 對地板施術（人已站在上面）= 包含活體 = 削弱
  - 對地板提前施術（人尚未進入）= 不包含活體 = 完整生效，人踩進去是物理結果
  - 對石頭施術讓石頭飛向人 = 石頭是作用對象 = 完整生效

### 7.4 場景設計原則
- Zone-based loading：玩家周圍一定半徑內為active physics zone，超出範圍變static mesh，destruction狀態persist
- 關卡式場景為主（效能考量），每個場景物件密度足以支撐物理玩法
- Destruction model：預設destruction為主，特定場景支援動態fracture

## 8. 經濟系統（暫定）

> 以下機制尚未確認是否納入，保留作為設計選項。

### 8.1 基本原則
- 無通貨膨脹設計：死亡清空裝備 = 資源持續從市場消失
- 稀有素材：expire date + 總量管控 + 實時drop rate微調

### 8.2 玩家職業生態
- 戰鬥型、製造型、物流型、情報型皆有生存空間
- 高端裝備需要玩家熟練度才能製造（NPC只能做基礎品）
- 物流業因稀有素材expire date而天然存在

### 8.3 死亡與資源流動
- 死亡清空裝備與熟練度，資源回流市場
- 新手永遠有機會，因為老手隨時可能翻車把資源吐回市場

## 9. 主線與敘事

### 9.1 玩家的敘事角色
- 見證者：原著主線照走，玩家是第六個從未被提及的人
- 玩家行為影響然與霆對自己的印象，不影響主線走向
- 主線劇情功能：填充世界觀與地區lore、觸發原著關鍵場景

### 9.2 地圖開放
- 不用主線任務鎖地圖
- 地圖開放由世界狀態驅動（勢力控制區、危險等級等）

### 9.3 支線結構
- NPC有機生活產生的自然故事 = 主要支線來源
- 無scripted quest清單，故事從NPC行為裡長出來
- 玩家撞進NPC的生活圈，自然捲入

### 9.4 動態支線任務系統

支線任務不是預設的任務清單，而是由NPC根據當下情況、與玩家的關係、以及自身需求動態生成。

**Reward Quota**

每位玩家在任何時間點都有一個當前可用的獎勵配額（Current Quota），上限由三個維度決定：

- 玩家等級（主要吃遊玩時間，類似原神冒險等階）
- 總技能熟練度累積值（死亡重置當前熟練度，但此處吃的是歷史總量，不受死亡影響）
- 地圖探索進度

這三個維度共同決定 **Absolute Maximum**。Current Quota低於Maximum時，隨時間緩慢回覆，上限為Maximum。Global market availability（當前世界裡未被採集、不在任何玩家視野內的野生素材總量）作為額外調節係數，影響實物獎勵的組成而非總價值。

坐著等刷新是最差的策略——等待期間探索地圖與練技能都在推高Maximum本身。

**任務生成流程**

當玩家與權重足夠高的NPC關係達到一定程度，或該NPC真實遭遇需要解決的問題時，NPC後台動態生成一個固定格式的任務JSON：

```json
{
  "type": "fetch",
  "number": 10,
  "object": "red berries",
  "reward": "<LLM從玩家當前quota中決定的子集，總價值與NPC權重成正比>",
  "player": "<player_id>",
  "report": "<npc_id>",
  "deadline": null,
  "deposit": null
}
```

```json
{
  "type": "find",
  "target": "<目標NPC或物件>",
  "reward": "<LLM決定的子集>",
  "player": "<player_id>",
  "report": "<npc_id>",
  "deadline": "<可選：限時任務>",
  "deposit": "<可選：NPC預付或要求玩家押金>"
}
```

JSON一旦由LLM生成並submit即鎖定，記入玩家狀態，直到任務完成、失敗或過期。任務難度與具體需求由LLM根據當下關係決定，不由玩家議價。

**後果與記憶**

任務過程與結果會被NPC記憶——完成品質、是否爽約、是否捲走押金，全部進入Civilizism的記憶與trust score系統。名聲可以臭掉，也可以建立起來。NPC之間會討論玩家的行為，這些資訊會透過正常的Civilizism社交傳播機制擴散。

### 9.5 玩家驅動力的入口分層

這個遊戲不預設單一驅動力。不同玩家會從不同入口進入並找到自己的重心：

小說黨的驅動力是與原著角色的真實互動——然與霆會記得你，這在任何其他媒介裡都不可能發生。強度黨的驅動力是本源體系的深度——沒有攻略，沒有上限，概念空間由玩家自己探索。世界黨的驅動力是地圖與lore密度——大量原著裡沒有篇幅放進去的背景敘事、勢力歷史、角色前傳，以碎片形式散落在世界裡等待被發掘。

對不熟悉原著的新玩家，主線是默認入口——跟著走就會開始理解這個世界，理解之後自然會找到自己想深入的那條線。

主線的功能之一是把人帶進坑，不是把人鎖在裡面。

## 10. 技術架構摘要

### 10.1 系統分層

| 層次 | 技術 | 說明 |
|------|------|------|
| NPC認知 | Civilizism（Python） | 記憶、人格、行為、計劃 |
| 能力審核 | LLM API | 節點理論可行性驗證，結果存檔後不再重複調用 |
| 術式執行 | Schema → Physics Engine | 確立後deterministic |
| 物理結算 | Godot/Unity Physics | 所有傷害與環境互動 |
| 渲染/遊戲邏輯 | Godot/Unity | 主引擎 |

### 10.2 LLM使用分層
- 主角團NPC：高品質API（GPT/Gemini等級）
- 重要模型NPC：中等模型
- 路人NPC：輕量本地模型或Civilizism的deterministic行為為主
- 術式審核：中等模型，結果存檔後不再重複調用

### 10.3 開發優先順序
1. Civilizism後端成熟（career project，並行進行）
2. 小說第一卷完成（IP基礎）
3. 物理引擎原型（核心玩法驗證）
4. 本源系統原型（LLM審核loop）
5. 整合與場景內容
